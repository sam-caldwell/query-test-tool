# Weight Calibration Methodology

## Overview

The calibration system empirically determines scoring weights by measuring the actual performance impact of SQL antipatterns against PostgreSQL via EXPLAIN ANALYZE.

The core insight: if we know which antipatterns are present in a query (via sqlscore) and we know the actual cost ratio between running that query on an optimized vs. degraded schema, we can use regression to determine how much each antipattern contributes to cost.

## Pipeline Architecture

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  Schema Gen │ →  │  Data Gen   │ →  │ Query Gen   │ →  │  EXPLAIN    │
│  10K schemas│    │  1K rows/tbl│    │  1M queries  │    │  Runner     │
└─────────────┘    └─────────────┘    └─────────────┘    └──────┬──────┘
                                                                 │
                                                                 ▼
                                                         ┌─────────────┐
                                                         │  OLS Ridge  │
                                                         │  Regression │
                                                         └─────────────┘
                                                                 │
                                                                 ▼
                                                         calibrated_weights.json
```

## Schema Generation

### Domain Archetypes (5 domains × ~2000 variants = 10,000 schemas)

| Domain | Tables | Description |
|--------|--------|-------------|
| E-commerce | users, products, orders, order_items, categories | Transactional workload |
| Blog | authors, posts, comments, tags, post_tags | Content platform |
| HR | departments, employees, salaries, projects, assignments | Enterprise |
| Inventory | warehouses, items, stock_levels, suppliers, purchase_orders | Supply chain |
| Analytics | users, sessions, events, pages, conversions | Event tracking |

### Mutation Types

Each mutation maps to one or more sqlscore rules:

| Mutation | Rules Tested | Effect |
|----------|-------------|--------|
| `drop_idx_{table}_{col}` | non-sargable, unbounded-sort | Remove single index |
| `drop_all_indexes` | non-sargable, unbounded-sort | Remove all non-PK indexes |
| `drop_fk_{table}_{col}` | missing-predicate | Remove foreign key |
| `widen_{table}_{N}` | select-star | Add N extra TEXT columns |
| `textify_{table}_{col}` | non-sargable | Change type to TEXT |
| `nullable_{table}` | non-sargable | Remove NOT NULL constraints |

### Variant Generation

Variants are generated by applying mutations in three tiers:
1. **Singles**: Every individual mutation (~50 per domain)
2. **Doubles**: Random pairs of mutations (sampled to fill target)
3. **Triples**: Random triples (fill remainder to hit 10K total)

## Query Generation

18 query templates cover all sqlscore rules:

| Template | Target Rule | Example |
|----------|------------|---------|
| SELECT * | select-star | `SELECT * FROM users LIMIT 100` |
| Non-sargable | non-sargable | `SELECT * FROM users WHERE LOWER(email) = 'test'` |
| Unbounded sort | unbounded-sort | `SELECT * FROM users ORDER BY name` |
| GROUP BY | group-by-fanout | `SELECT status, COUNT(*) FROM orders GROUP BY status` |
| Window (no partition) | window-function | `SELECT id, row_number() OVER (ORDER BY id) FROM users` |
| EXISTS subquery | correlated-subquery | `SELECT * FROM users WHERE EXISTS (...)` |
| DISTINCT + JOIN | distinct-dedup | `SELECT DISTINCT u.* FROM users u JOIN orders o ON ...` |
| Cartesian | cartesian-product | `SELECT * FROM users, orders LIMIT 100` |
| UNION | set-operation | `SELECT name FROM users UNION ALL SELECT name FROM users` |
| CTE | cte | `WITH cte AS (...) SELECT * FROM cte` |
| Boolean nesting | boolean-nesting | `SELECT * FROM t WHERE (a AND b) OR (c AND d)` |
| CASE | case-expression | `SELECT CASE WHEN ... END FROM t` |

Each template is parameterized with actual table/column names from the domain.

## EXPLAIN Execution

For each (query, schema_instance) pair:
1. Set `search_path` to the schema
2. Execute `EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)`
3. Parse plan JSON for: Total Cost, Actual Time, Rows, Buffer hits/reads
4. Score the query with sqlscore
5. Store both EXPLAIN metrics and sqlscore findings

Execution is concurrent (configurable workers, default 8) with per-query timeouts.

## Regression Model

### Setup

For each query, we compare the degraded schema cost against the optimal schema cost:

```
cost_ratio = degraded_cost / optimal_cost
```

The regression model:

```
log(cost_ratio) = β₀ + Σ βᵢ × count(rule_i in findings)
```

We use log(cost_ratio) because performance degradation is multiplicative, not additive.

### Ridge Regularization

Since many features are sparse (a given query only triggers 1-3 rules), we add Tikhonov regularization (ridge) with λ = 0.01 to prevent singularity:

```
β = (X^T X + λnI)^{-1} X^T y
```

### Weight Extraction

The regression coefficients βᵢ represent the log-scale cost multiplier per occurrence of each rule. We scale these to user-friendly penalty weights:

```
weight_i = max(0, βᵢ × 10)
```

Negative coefficients are clamped to zero (an antipattern shouldn't reduce the score).

### Validation

The output includes:
- **R²**: Coefficient of determination (model fit quality)
- **Per-weight validation**: flags unusually high weights or too many zeros
- **Sample size**: number of data points used

## Schema Tracking Database

All artifacts are tracked in a `calibration` PostgreSQL schema:

```sql
calibration.families          -- Domain families
calibration.schema_instances  -- 10K schema variants with DDL and mutation metadata
calibration.queries           -- 1M generated queries with target rules
calibration.results           -- EXPLAIN results + sqlscore findings
```

This enables:
- Resumable execution (skip already-completed work)
- Post-hoc analysis (query the results directly)
- Iterative refinement (re-run regression with different filtering)

## Practical Considerations

### Time Estimates

| Phase | ~Duration | I/O |
|-------|-----------|-----|
| Schema generation | 10-30 min | DDL + INSERT for 10K schemas |
| Query generation | 1-2 min | 1M INSERT into tracking table |
| EXPLAIN execution | 30-60 min | 2M EXPLAIN calls at ~1ms each |
| Regression | < 1 sec | Pure computation |

### Resource Requirements

- PostgreSQL: ~2GB disk for 10K schemas with 1K rows/table
- Memory: ~500MB for the Go process during EXPLAIN phase
- CPU: Scales linearly with `-workers` flag

### Tuning

- **Fewer schemas**: Use `-schemas 1000` for a quick calibration run
- **Fewer queries**: Use `-queries 100000` for faster iteration
- **More data**: Use `-rows 10000` for more realistic cost differentiation (slower)
- **More workers**: Use `-workers 16` on machines with many cores and fast disk

## Interpreting Results

A well-calibrated result should show:
- **R² > 0.5**: The model explains at least half the cost variance
- **Positive weights**: All rules have non-negative penalties
- **Relative ordering**: cartesian-product > correlated-subquery > non-sargable > select-star (expected from theory)
- **Zero weights**: Some cognitive rules (cte, case-expression) may be near-zero since they don't affect EXPLAIN cost — this is correct for the efficiency dimension but should be supplemented with human readability assessments for cognitive weights
